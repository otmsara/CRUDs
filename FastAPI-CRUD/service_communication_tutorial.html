Chapter 1 · Concept

How the Two Services Communicate

The auth and management services never call each other directly. There is no HTTP request from one to the other on every API call. They communicate through a shared secret and a signed token — the JWT.

The Big Picture

Here is the full communication flow when a supervisor logs in and submits a passenger list:

Angular
Frontend
POST /auth/login
Port 8001
Auth Service
JWT token
Angular
Frontend
Bearer token
Port 8002
Management

Notice that the auth service and the management service never speak to each other. The frontend is the courier — it gets a token from auth, then carries it to management. Management verifies the token using the same secret key that auth used to sign it.

Why No Direct HTTP Calls Between Services?

A naive approach would be: management receives a request → calls auth service at GET /auth/me to verify the token → auth responds with user info. This works but creates tight coupling and a network call on every single request. If the auth service is slow or down, every management endpoint breaks.

The JWT approach is stateless: the token itself carries all the information management needs (user ID, role). Management can verify it locally in microseconds using the shared secret — no network call, no dependency at runtime.

Key Rule
The only thing shared between the two services is the JWT_SECRET environment variable. That one string is what makes the whole system work. Auth uses it to sign tokens. Management uses it to verify them. They never need to call each other.

The Two Databases Are Independent

Your auth service uses auth_db. Your management service uses management_db. They are two separate PostgreSQL databases. The User table in management is not the same table as the one in auth — it is a separate copy maintained by the management service, created by the admin when they add users via POST /users/.

Important
This means when you create a user in the auth service (register), that user does not automatically appear in the management service's database. In your project, user creation happens via the admin endpoint POST /users/ on the management service — which creates the user in management_db and hashes the password there. The auth service then uses auth_db separately for login. In a more mature system you would sync these via an event — but for your internship scope, the admin creates users in both or you use one shared DB.

Practical Recommendation For Your Project

Given the 3.5-month timeline and that both services are in the same Docker Compose, the simplest working approach is: use one shared PostgreSQL database with two schemas, or simply one database that both services point to. This way a user registered via auth is immediately visible to management. The JWT bridge still works identically — the secret is still shared, the token is still verified locally. You eliminate the sync problem entirely.

To do this, set both services' DATABASE_URL to the same PostgreSQL connection string in .env:

.env (root)
# Both services share one database — simpler for your project scope
AUTH_DATABASE_URL=postgresql+asyncpg://postgres:postgres@localhost:5433/transport_db
MGMT_DATABASE_URL=postgresql+asyncpg://postgres:postgres@localhost:5433/transport_db

# The one shared secret — MUST be identical in both services
JWT_SECRET=your-very-long-random-secret-minimum-32-chars
JWT_ALGORITHM=HS256

Chapter 2 · Concept

The JWT Bridge

A JWT is a self-contained signed envelope. It carries the user's identity and role, cryptographically sealed so no one can tamper with it. The management service cracks it open and trusts what it finds inside.

What a JWT Actually Looks Like

A JWT is three base64url-encoded parts separated by dots: header.payload.signature. When you paste your token into jwt.io, you see something like this:

Decoded JWT payload (what management reads)
{
  "sub":  "a3f8c2d1-4b5e-4f6a-8c9d-1e2f3a4b5c6d",  ← user UUID
  "role": "supervisor",                           ← role embedded by auth
  "type": "access",                               ← must be "access", not "refresh"
  "exp":  1714000000,                              ← expiry timestamp
  "iat":  1713999100                               ← issued-at timestamp
}

The management service reads sub to know who is calling, and role to know what they're allowed to do. It never needs to query the auth service — everything it needs is right there in the token.

How the Signature Prevents Tampering

The signature is computed as HMAC-SHA256(base64(header) + "." + base64(payload), JWT_SECRET). If anyone changes a single character in the payload — say, changing "role": "employee" to "role": "admin" — the signature no longer matches, and the management service immediately rejects the token with a 401.

Never
Never send the JWT_SECRET to the frontend, expose it in logs, or commit it to Git. Anyone who has it can forge tokens with any role they want. Store it only in .env (which is in .gitignore) and Docker environment variables.

Access Token vs Refresh Token

Your auth service issues two tokens. Only the access token should ever be sent to the management service. The management service explicitly rejects refresh tokens by checking the "type" claim:

Token type check in dependencies.py
# In management service dependencies.py
token_type: str = payload.get("type")
if token_type != "access":
    raise credentials_exception   ← rejects refresh tokens

Chapter 3 · Auth Service Side

Embedding the Role in the Token

The auth service must embed the user's role in the JWT payload when it creates the token. Without this, the management service would have to query the database on every request to find the role — defeating the whole point.

The Problem with Your Current Auth Service

Your auth service tutorial created the JWT utility in app/utils/jwt.py with this signature:

AUTHapp/utils/jwt.py — original
def create_access_token(user_id: int) -> str:
    payload = {
        "sub":  str(user_id),
        "type": "access",
        "exp":  expire,
        "iat":  datetime.now(timezone.utc)
    }
    return jwt.encode(payload, settings.JWT_SECRET, algorithm=settings.JWT_ALGORITHM)

This only puts user_id in the token. The management service has no way to know the user's role without a DB query. We need to add role to the payload.

Updated JWT Utility — Add role Parameter

Open services/auth/app/utils/jwt.py and update create_access_token:

AUTHapp/utils/jwt.py — updated
from jose import jwt
from datetime import datetime, timedelta, timezone
from app.config import settings

def create_access_token(user_id: str, role: str) -> str:
    """
    Create a short-lived access token.
    role is embedded so management can read it without a DB call.
    """
    expire = datetime.now(timezone.utc) + timedelta(
        minutes=settings.ACCESS_TOKEN_EXPIRE_MINUTES
    )
    payload = {
        "sub":  str(user_id),
        "role": role,          ← NEW: embed role here
        "type": "access",
        "exp":  expire,
        "iat":  datetime.now(timezone.utc)
    }
    return jwt.encode(payload, settings.JWT_SECRET, algorithm=settings.JWT_ALGORITHM)

def create_refresh_token(user_id: str) -> str:
    """Refresh tokens do NOT carry the role — they only identify the user."""
    expire = datetime.now(timezone.utc) + timedelta(
        days=settings.REFRESH_TOKEN_EXPIRE_DAYS
    )
    payload = {
        "sub":  str(user_id),
        "type": "refresh",
        "exp":  expire,
        "iat":  datetime.now(timezone.utc)
    }
    return jwt.encode(payload, settings.JWT_SECRET, algorithm=settings.JWT_ALGORITHM)

def decode_token(token: str) -> dict | None:
    try:
        payload = jwt.decode(
            token, settings.JWT_SECRET,
            algorithms=[settings.JWT_ALGORITHM]
        )
        return payload
    except JWTError:
        return None

Update the Login Service to Pass the Role

Now update services/auth/app/services/auth_service.py — the login_user function — to pass the user's role when calling create_access_token:

AUTHapp/services/auth_service.py — login_user, updated lines
async def login_user(data: LoginRequest, db: AsyncSession):
    # ... (verify password logic unchanged) ...

    # BEFORE (wrong — no role in token):
    # access_token = create_access_token(user.id)

    # AFTER (correct — role is passed in):
    access_token  = create_access_token(
        user_id=str(user.id),
        role=user.role.value   ← .value converts enum to string: "admin", "supervisor", "employee"
    )
    refresh_token = create_refresh_token(user_id=str(user.id))

    # ... (persist refresh token, return response — unchanged) ...
Why .value
Your Role is a Python Enum. user.role gives you Role.supervisor. The JWT needs a plain string, not an enum object. Always call .value before embedding in the payload: user.role.value → "supervisor".

Verify It Works

Log in via Postman with a supervisor account. Copy the access_token and paste it into jwt.io. In the decoded payload section you should now see:

jwt.io — decoded payload
{
  "sub":  "a3f8c2d1-4b5e-4f6a-8c9d-1e2f3a4b5c6d",
  "role": "supervisor",
  "type": "access",
  "exp":  1714000000,
  "iat":  1713999100
}

If you see "role" in that payload, the auth service side is complete. Now onto the management service.

Chapter 4 · Management Service Side

Decoding & Trusting the Token

The management service needs the exact same JWT secret as the auth service. It uses it to verify the signature — if the token passes verification, the payload is trusted completely. No auth service call, no DB lookup for the role.

Step 1 — Share the Secret via Environment Variable

Both services must have the same JWT_SECRET value. Open your .env file at the root of your project (or the one in each service folder) and make sure it is identical:

services/auth/.env
DATABASE_URL=postgresql+asyncpg://postgres:postgres@localhost:5433/transport_db
JWT_SECRET=transport-system-super-secret-key-minimum-32-chars
JWT_ALGORITHM=HS256
ACCESS_TOKEN_EXPIRE_MINUTES=15
REFRESH_TOKEN_EXPIRE_DAYS=7
# ... other auth-specific vars (MAIL_*, REDIS_URL, etc.) ...
services/management/.env
DATABASE_URL=postgresql+asyncpg://postgres:postgres@localhost:5433/transport_db
JWT_SECRET=transport-system-super-secret-key-minimum-32-chars  ← IDENTICAL
JWT_ALGORITHM=HS256
Critical
If the secrets are different — even by one character — the management service will reject every single token with a 401 Unauthorized. This is the most common mistake when setting up service communication for the first time. Double-check both .env files are in sync.

Step 2 — Management config.py Reads the Secret

Your management service's config.py already declares JWT_SECRET. Confirm it looks like this:

MGMTapp/config.py
from pydantic_settings import BaseSettings
from functools import lru_cache

class Settings(BaseSettings):
    DATABASE_URL:  str
    JWT_SECRET:    str           ← reads from .env, same value as auth
    JWT_ALGORITHM: str = "HS256"  ← must match auth service

    class Config:
        env_file = ".env"

@lru_cache
def get_settings() -> Settings:
    return Settings()

settings = get_settings()

Step 3 — What Happens When Management Decodes a Token

The jose library does the following when you call jwt.decode():

  1. Splits the token into header, payload, and signature parts.
  2. Re-computes the expected signature using JWT_SECRET and the header + payload.
  3. Compares the computed signature with the one in the token. If they differ → JWTError → 401.
  4. Checks the exp claim against the current time. If expired → JWTError → 401.
  5. If all checks pass → returns the payload dict. Management trusts it completely.
MGMTWhat jose.jwt.decode() does internally
# This one line does ALL of the above checks:
payload = jwt.decode(
    token,
    settings.JWT_SECRET,           ← same secret auth used to sign
    algorithms=[settings.JWT_ALGORITHM]
)

# If this line doesn't raise → token is valid, not tampered, not expired
# payload is now a trusted dict: {"sub": "uuid", "role": "supervisor", ...}
user_id = payload["sub"]    ← safe to trust
role    = payload["role"]   ← safe to trust

Chapter 5 · Management Service Side

The get_current_user Dependency

This is the bridge between the raw JWT in the request header and the User object your endpoints work with. Every protected endpoint in the management service goes through this single function.

The Complete dependencies.py File

This is the file that makes everything work together. Read every comment — each line has a specific reason.

MGMTapp/dependencies.py — complete file
from fastapi import Depends, HTTPException, status
from fastapi.security import OAuth2PasswordBearer
from jose import jwt, JWTError
from sqlalchemy.ext.asyncio import AsyncSession
from sqlalchemy import select
from app.config import settings
from app.database import get_db
from app.models import User, Role

# Tells FastAPI where login happens — used only for the /docs Authorize button UI
# The actual token validation is done manually below, not by OAuth2PasswordBearer itself
oauth2_scheme = OAuth2PasswordBearer(
    tokenUrl="http://localhost:8001/auth/login"   ← points to auth service
)

async def get_current_user(
    token: str = Depends(oauth2_scheme),      ← extracts Bearer token from header
    db: AsyncSession = Depends(get_db)
) -> User:
    """
    Decodes the JWT from the Authorization header.
    Returns the User ORM object if the token is valid.
    Raises 401 if anything is wrong with the token.
    """
    credentials_exception = HTTPException(
        status_code=status.HTTP_401_UNAUTHORIZED,
        detail="Could not validate credentials",
        headers={"WWW-Authenticate": "Bearer"},
    )

    try:
        # Step 1: Decode and verify the JWT using the shared secret
        payload = jwt.decode(
            token,
            settings.JWT_SECRET,
            algorithms=[settings.JWT_ALGORITHM]
        )

        # Step 2: Extract the user_id from the "sub" claim
        user_id: str = payload.get("sub")
        if user_id is None:
            raise credentials_exception

        # Step 3: Reject refresh tokens — only access tokens are accepted here
        token_type: str = payload.get("type")
        if token_type != "access":
            raise credentials_exception

    except JWTError:
        # Covers: invalid signature, expired token, malformed token
        raise credentials_exception

    # Step 4: Load the user from management DB to get fresh is_active status
    result = await db.execute(select(User).where(User.id == user_id))
    user = result.scalars().first()

    if user is None:
        raise credentials_exception   ← user was deleted after token was issued

    if not user.is_active:
        raise HTTPException(
            status_code=status.HTTP_403_FORBIDDEN,
            detail="Account is deactivated"
        )

    return user   ← the full User ORM object, ready to use in endpoints


def require_role(*roles: Role):
    """
    Factory function that returns a dependency checking the user's role.
    Usage: Depends(require_role(Role.admin))
           Depends(require_role(Role.admin, Role.supervisor))
    """
    async def _check(current_user: User = Depends(get_current_user)) -> User:
        if current_user.role not in roles:
            raise HTTPException(
                status_code=status.HTTP_403_FORBIDDEN,
                detail=f"Required role: {[r.value for r in roles]}"
            )
        return current_user
    return _check

Why We Still Query the DB in Step 4

The role and user_id from the JWT are trusted. But what about is_active? If an admin deactivates a user, their existing token is still cryptographically valid until it expires (up to 15 minutes). The DB query catches this — we check user.is_active against the live database on every request.

Note
This is one of the few cases where we do query the DB during token verification — but it is querying the management service's own database, not calling the auth service. The round-trip stays entirely within the management service.

What OAuth2PasswordBearer Actually Does

It is a FastAPI utility that extracts the token from the Authorization: Bearer <token> header. That is its only job. It does not validate the token, does not call the auth service, and does not check roles. All of that is done in the code above it. The tokenUrl parameter is only used to make the /docs Authorize button point to the right login URL.

Chapter 6 · Management Service Side

Role Guards in Action

With get_current_user and require_role in place, protecting any endpoint is a single line. Here is how it looks in each of your three routers with real examples from your Transport System.

Pattern A — Admin Only

Used in routers/users.py. No user except an admin should be able to create or delete users.

MGMTapp/routers/users.py — admin-only example
from app.dependencies import require_role
from app.models import Role

# Pre-build the dependency once at module level — cleaner than repeating it
AdminOnly = Depends(require_role(Role.admin))

@router.post("/", response_model=UserRead, status_code=201)
async def create_user(
    data: UserCreate,
    db: AsyncSession = Depends(get_db),
    _: User = AdminOnly          ← underscore: we don't need the user object, just the check
):
    return await user_service.create_user(data, db)

@router.delete("/{user_id}", status_code=204)
async def delete_user(
    user_id: UUID,
    db: AsyncSession = Depends(get_db),
    _: User = AdminOnly
):
    await user_service.delete_user(user_id, db)

Pattern B — Multiple Roles Allowed

Used in routers/passengers.py. Both admin and supervisor can manage passengers.

MGMTapp/routers/passengers.py — multi-role example
AdminOrSupervisor = Depends(require_role(Role.admin, Role.supervisor)) @router.get("/", response_model=List[PassengerRead]) async def list_passengers( db: AsyncSession = Depends(get_db), current_user: User = AdminOrSupervisor ← we DO need the user here (for scoping) ): # Pass current_user to service — supervisor only sees their passengers return await passenger_service.get_passengers(db, current_user)

Pattern C — Any Authenticated User

Used in routers/requests.py. Any logged-in user can create a change request or view their own.

MGMTapp/routers/requests.py — any-auth example
from app.dependencies import get_current_user   ← no role check, just authentication

@router.post("/{passenger_id}", response_model=ChangeRequestRead, status_code=201)
async def create_request(
    passenger_id: UUID,
    data: ChangeRequestCreate,
    db: AsyncSession = Depends(get_db),
    current_user: User = Depends(get_current_user)   ← any role, just must be logged in
):
    return await request_service.create_request(passenger_id, data, current_user, db)

What Happens When a Wrong Role Calls an Endpoint

If a supervisor calls DELETE /users/{id} which requires admin:

  1. FastAPI receives the request and sees the endpoint has AdminOnly dependency.
  2. FastAPI runs get_current_user → decodes JWT → loads User from DB → returns the supervisor User object.
  3. FastAPI runs require_role(Role.admin) → checks user.role → it's Role.supervisor → not in (Role.admin,).
  4. Raises HTTPException(403) with detail: "Required role: ['admin']".
  5. The actual delete_user function body is never executed.
Clean Code
Define role dependencies as module-level constants (AdminOnly, AdminOrSupervisor) at the top of each router file. This avoids repeating Depends(require_role(Role.admin)) on every endpoint and makes the permission model readable at a glance.

Quick Debug Checklist

When a call between services breaks, check these in order:

  1. Paste the token in jwt.io. Does the payload contain "role"? If not → auth service not updated.
  2. Check both .env files. Is JWT_SECRET exactly the same string in both? Copy-paste, don't retype.
  3. Is the token expired? Check the exp field in jwt.io — it's a Unix timestamp. Compare to current time.
  4. Are you sending the access_token and not the refresh_token? They look identical but the payload "type" field differs.
  5. Is the user in the management database? If you only registered via the auth service but never called POST /users/ on management, the management DB won't find the user → 401.
